## 1. Scope and how to read this plan

- **The deliverable is the numbered task table in ¬ß3.** Tasks are ordered so that
  every task's dependencies appear earlier; the `Depends on` column lists hard
  prerequisites (tasks with disjoint dependencies can run in parallel).
- Each task is roughly **one PR** in its primary repo. Where a task necessarily
  spans two repos (e.g. a DBOS workflow plus its API enqueue endpoint, or an API
  endpoint plus its DTO schemas), the `Repo` column lists both; submodule bumps
  ride along.
- **API DTO Zod schemas** (design-delta ¬ß2.11, last row) are added to
  `database-lib` incrementally *by the API/UI task that introduces the endpoint* ‚Äî
  the DTO's failing parse/reject tests are written in `database-lib` first, then
  the endpoint is built against them. They are not one big up-front task.
- Repo abbreviations: **db-lib** = `supagloo-database-lib`, **api** =
  `supagloo-nodejs-api`, **dbos** = `supagloo-nodejs-dbos`, **nextjs** =
  `supagloo-nextjs`, **root** = `supagloo` (Compose orchestration).

### Test strategy (applies to every task ‚Äî stated once, not repeated per row)

**TDD rhythm, no exceptions:** every task starts **red** ‚Äî write the failing unit
tests *and* the failing e2e spec that capture the intended behavior first ‚Äî then
implements the minimum to go **green**, then refactors with the tests as the
safety net. Nothing merges red. The `Unit tests` / `E2E tests` columns state
*what the tests verify*, not test code.

**Unit tests everywhere:** every task has unit tests (Vitest in every repo,
matching the existing `supagloo-nextjs` setup) ‚Äî pure logic in isolation, network
and DB mocked/faked.

**What "e2e" means per layer:**

| Layer | E2E definition |
|---|---|
| **db-lib** | Apply real migrations to the Compose Postgres and exercise the generated Prisma client against it: constraint behavior (uniques, composites, FKs), JSON-column round-trips, enum values. No mocks. |
| **api** | Real HTTP calls against a running API instance backed by the Compose Postgres + MinIO. OpenRouter/Gloo/YouVersion egress hits the **real provider APIs** (design-delta ¬ß10); GitHub egress uses the task-9 github-stub + local git server. No in-process mocking. |
| **dbos** | Actually enqueue via `DBOSClient` (workflowID = domain-record id) and run the workflow to completion against the real `supagloo_dbos` system DB + app DB + MinIO. Includes **crash/replay tests** where the design emphasizes recovery: kill the worker mid-workflow, restart, assert completed steps are not re-executed and side effects are not duplicated. |
| **nextjs (UI)** | **Stagehand** specs (existing harness in `tests/e2e/`) driven against the full Compose stack. The existing flag-gated demo/test-session seam is extended (task 23) to obtain a *real* session via the flag-gated `POST /v1/test/seed` endpoint instead of faking browser state ‚Äî so UI e2e exercises browser ‚Üí BFF ‚Üí API ‚Üí Postgres/MinIO/DBOS for real. Existing mock-session-only specs remain for pure-UI regressions. Playwright is not introduced; Stagehand is the established UI-e2e tool here, and non-UI e2e never uses a browser. |

**Provider egress in e2e** *(policy revised 2026-07-22 by design-delta ¬ß10 ‚Äî
supersedes the original "live-provider smoke tests are manual/optional, never
gating" line for OpenRouter/Gloo/YouVersion)*: for **OpenRouter, Gloo, and
YouVersion**, e2e always exercises the **real provider APIs** ‚Äî real-provider
e2e *is* the gating suite. An e2e test either hits the real provider or does
not exercise that provider at all; deterministic provider *misbehavior*
(injected failures, controlled timing) is a **unit** concern via injected
fetch, never an e2e one (¬ß10.6). Required real secrets come from `.env`
(tasks 34-E2/34-E8); a missing secret **fails e2e setup fast** rather than
silently skipping. The sole exception: interactive browser logins (YouVersion
OAuth sign-in, OpenRouter's PKCE page) cannot be automated with static
credentials ‚Äî UI specs may shim *only that interactive hop* (¬ß10.2/¬ß10.4b).
**GitHub is out of scope for this policy** and keeps the original round-1
posture: env-overridable base URLs pointed at the github-stub + local git
smart-HTTP server (task 9), so clone/push/PR flows run against real git,
deterministically.

## 2. Binding constraints honored throughout (from `.claude/agents/tech-lead-memory/`)

- **Static DBOS workflows only**; API enqueues via `DBOSClient` with
  `workflowID` = domain-record id; status flows back via app-DB rows; no HTTP
  between API and DBOS.
- **GitHub App** (not OAuth app): only `installationId` stored; short-lived
  installation tokens minted on demand; every git-ops workflow starts with
  `mintInstallationToken`.
- **Composition source of truth is the repo** (`supagloo.project.json`, Zod-
  validated); no Composition/Scene tables; media in S3 by key.
- **KJV/BSB only** for the generation pipeline; version ids resolved via the
  YouVersion "Get a Bible collection" endpoint at implementation time ‚Äî never
  hardcoded.
- **OpenRouter covers all media modalities**; AI SDK `generateObject` for
  structured text, plain `fetch` for media; **model ids never hardcoded**
  (discovery endpoints).
- **MinIO local S3 parity** with dual endpoints (`S3_ENDPOINT` /
  `S3_PUBLIC_ENDPOINT`); one Postgres 17 container, two logical DBs.
- **Prisma exact-version pin** in api/dbos, CI-enforced against db-lib's
  exported pin.
