# Supagloo — TDD Implementation Plan

*Written 2026-07-17 as Step 5 of the `/design` workflow. Implements everything in
[`design-delta.md`](./design-delta.md) starting from the current state described in
[`current-design.md`](./current-design.md) (real mocked-frontend Next.js app; empty
`database-lib` / `api` / `dbos` scaffolds; Compose runs only the UI). This is a
planning document — no code is written by this step. **Requires user review and
approval before implementation begins.***

---

## 1. Scope and how to read this plan

- **The deliverable is the numbered task table in §3.** Tasks are ordered so that
  every task's dependencies appear earlier; the `Depends on` column lists hard
  prerequisites (tasks with disjoint dependencies can run in parallel).
- Each task is roughly **one PR** in its primary repo. Where a task necessarily
  spans two repos (e.g. a DBOS workflow plus its API enqueue endpoint, or an API
  endpoint plus its DTO schemas), the `Repo` column lists both; submodule bumps
  ride along.
- **API DTO Zod schemas** (design-delta §2.11, last row) are added to
  `database-lib` incrementally *by the API/UI task that introduces the endpoint* —
  the DTO's failing parse/reject tests are written in `database-lib` first, then
  the endpoint is built against them. They are not one big up-front task.
- Repo abbreviations: **db-lib** = `supagloo-database-lib`, **api** =
  `supagloo-nodejs-api`, **dbos** = `supagloo-nodejs-dbos`, **nextjs** =
  `supagloo-nextjs`, **root** = `supagloo` (Compose orchestration).

### Test strategy (applies to every task — stated once, not repeated per row)

**TDD rhythm, no exceptions:** every task starts **red** — write the failing unit
tests *and* the failing e2e spec that capture the intended behavior first — then
implements the minimum to go **green**, then refactors with the tests as the
safety net. Nothing merges red. The `Unit tests` / `E2E tests` columns state
*what the tests verify*, not test code.

**Unit tests everywhere:** every task has unit tests (Vitest in every repo,
matching the existing `supagloo-nextjs` setup) — pure logic in isolation, network
and DB mocked/faked.

**What "e2e" means per layer:**

| Layer | E2E definition |
|---|---|
| **db-lib** | Apply real migrations to the Compose Postgres and exercise the generated Prisma client against it: constraint behavior (uniques, composites, FKs), JSON-column round-trips, enum values. No mocks. |
| **api** | Real HTTP calls against a running API instance backed by the Compose Postgres + MinIO (and the provider-stub servers of task 9 for GitHub/OpenRouter/Gloo/YouVersion egress). No in-process mocking. |
| **dbos** | Actually enqueue via `DBOSClient` (workflowID = domain-record id) and run the workflow to completion against the real `supagloo_dbos` system DB + app DB + MinIO. Includes **crash/replay tests** where the design emphasizes recovery: kill the worker mid-workflow, restart, assert completed steps are not re-executed and side effects are not duplicated. |
| **nextjs (UI)** | **Stagehand** specs (existing harness in `tests/e2e/`) driven against the full Compose stack. The existing flag-gated demo/test-session seam is extended (task 23) to obtain a *real* session via the flag-gated `POST /v1/test/seed` endpoint instead of faking browser state — so UI e2e exercises browser → BFF → API → Postgres/MinIO/DBOS for real. Existing mock-session-only specs remain for pure-UI regressions. Playwright is not introduced; Stagehand is the established UI-e2e tool here, and non-UI e2e never uses a browser. |

**Provider egress in e2e** is deterministic: all outbound provider base URLs
(GitHub API, OpenRouter, Gloo, YouVersion) are env-overridable and pointed at the
provider-stub harness (task 9), which includes a local git smart-HTTP server so
clone/push/PR flows run against real git. Live-provider smoke tests are manual/
optional, never part of the gating suite.

## 2. Binding constraints honored throughout (from `.claude/agents/tech-lead-memory/`)

- **Static DBOS workflows only**; API enqueues via `DBOSClient` with
  `workflowID` = domain-record id; status flows back via app-DB rows; no HTTP
  between API and DBOS.
- **GitHub App** (not OAuth app): only `installationId` stored; short-lived
  installation tokens minted on demand; every git-ops workflow starts with
  `mintInstallationToken`.
- **Composition source of truth is the repo** (`supagloo.project.json`, Zod-
  validated); no Composition/Scene tables; media in S3 by key.
- **KJV/BSB only** for the generation pipeline; version ids resolved via the
  YouVersion "Get a Bible collection" endpoint at implementation time — never
  hardcoded.
- **OpenRouter covers all media modalities**; AI SDK `generateObject` for
  structured text, plain `fetch` for media; **model ids never hardcoded**
  (discovery endpoints).
- **MinIO local S3 parity** with dual endpoints (`S3_ENDPOINT` /
  `S3_PUBLIC_ENDPOINT`); one Postgres 17 container, two logical DBs.
- **Prisma exact-version pin** in api/dbos, CI-enforced against db-lib's
  exported pin.

## 6. Risks & notes flagged (no plan deviations — sign-off items)

1. **Provider-stub harness (task 9)** is plan-added test infrastructure the
   design implies but doesn't specify (env-overridable provider base URLs, a
   local git smart-HTTP server). Without it, deterministic e2e of git-ops and
   AI/media workflows is impossible. It never runs in production images.
2. **Seed endpoint scheduled early (task 10, M2)** rather than under final
   hardening: every later e2e task depends on it, and §9-Q9 already approved it.
3. **Shared-helper placement (tasks 11, 13, 32/34/36):** two families of
   helpers are used by **both** the API and DBOS and are recommended to live in
   `database-lib` (precedent: the shared crypto helpers) so there is one
   implementation, not two:
   - **`mintInstallationToken`** (App JWT signing + installation-token
     exchange) — the design text has each service minting its own; functionally
     identical either way.
   - **S3 key-layout helpers** (`projects/{id}/assets/{assetId}`,
     `renders/{jobId}/output.mp4|thumb.jpg`) — needed by both the API (task 13)
     and DBOS (tasks 32/34/36); a single shared helper prevents key-format
     drift between the writer (DBOS) and the presigner (API).

   Confirm both placements at task 11/13. **Note:** item #1's create-new-repo
   **user-token** exchange (JIT zero-storage hop) is **API-only** — there is no
   user context in the DBOS scaffold workflow — so it is deliberately **not**
   part of this shared-helper item.
4. **Render e2e is genuinely slow** (real Chromium/Remotion in task 36/38/47).
   These specs are tagged and run in a separate heavy lane, not on every unit
   push — the gating principle (never merge red) still applies to the lane.
5. **Railway `CREATE DATABASE` check (§9-Q7b)** lands in task 46 with the
   documented schema-isolation fallback — per the design, an implementation-time
   verification, not a blocker.
